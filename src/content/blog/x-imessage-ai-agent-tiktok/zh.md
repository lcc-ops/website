据报一条工作流:中国侧用户发一行 iMessage 给一个美国-resident AI agent。agent 负责 TikTok 注册、滚动十条热门视频、逐条做摘要、再把综合摘要回发回中国。用户没装 TikTok,没接 VPN,也没碰任何美国 IP。经济学有意思的地方不是 AI 本身——而是用户和 agent 之间的角色切分,以及每侧承担的成本栈。

## 流水线逐步拆开

| 步骤 | 跑在哪 | 输入 | 输出 |
|---|---|---|---|
| 用户发 iMessage | 中国 | 一句短文,例如"trending now" | 出站触发 |
| Agent 任务启动 | 美国(residential IP) | 入站 iMessage | 新 TikTok 账号、登录会话 |
| 账号注册 | 美国 | 电话或邮箱、年龄确认 | 活跃会话 cookie |
| Trending 拉取 | 美国 | 会话 cookie | 10 个视频 ID 和字幕 |
| 单条摘要 | 美国(LLM) | 字幕 | 1–2 句概要 |
| 综合消息编排 | 美国(LLM + agent) | 10 条概要 | 一段合并消息 |
| iMessage 回送 | 中国 | 合并消息 | 用户 1–3 分钟读到 |

每步都小。首尾连起来,这回路给到用户的是他们自己用 30 分钟也做不出来的一本摘要。

## 成本实际落点不在 AI

| 成本项 | 每次成本(估计) | 说明 |
|---|---:|---|
| TikTok 账号注册 | 0.00–0.20 美元 | 电话号、验证码 |
| 美国住宅代理 | 0.20–0.50 美元 | 单次会话,5–15 分钟 |
| LLM 摘要,10 条 | 0.05–0.15 美元 | Haiku 级别,约 3k 入、1k 出 |
| iMessage 中转 | 0.00–0.05 美元 | 短信网关成本 |
| 调度开销 | 0.10–0.30 美元 | 重试、字段校验 |
| 单次合计 | 0.40–1.20 美元 | 远低于实习生小时费 |

同样的活让美国实习生干,freelance 价一般 25–60 美元。差价不是"AI 替代 AI"——是"远程 agent 用十分之一的价替代低频 gig 劳动"。这才是背后的模式,不是 AI 本身。

## 用户侧比想象中难

用户那条 iMessage 也不是零成本。摩擦层叠加:

1. 要先知道有这条 iMessage 通道(选通道)。
2. 要把请求格式化成 agent 能解的样子。
3. 要等 1–10 分钟往返。
4. 要相信摘要够新(trending 几分钟就变)。
5. 要付 agent 后面那位 operator 的钱。

对一次性使用,用户花的 setup 心力比省下的多。成本优势在每周跑几次才显出来,setup 才能分摊。

## AI 改了什么

agent 流水线干了四件窄活:

- 解析入站 iMessage 成结构化任务。
- 浏览器自动化跑注册和滚动。
- 对每条字幕跑 LLM。
- 把摘要粘合成一条消息。

这四件在 2026 年都不新鲜。新鲜的是打包成一条跨中美通道的聊天分发回路。AI 组件全是现成的,价值在调度层——它处理住宅代理、账号指纹漂移、验证码 fallback 和消息回拼。

调度层也是护城河。另一个 operator 一个周末就能抄走这个栈。抄不走的是几个月积累下来的失败处理——TikTok 丢掉住宅 IP 时怎么办、SMS 服务商限流时怎么办、跨时区 iMessage 延迟时怎么办。这些失败路径没文档,只在 operator 的日志里。

## 案例没盖的四件

推文留下四个口子。

**养号。** 全新 TikTok 账号看到的 trending feed 和老号不一样。trending 同时按账号年龄和兴趣权重。若不先养几轮滚动和点赞,摘要会偏向低互动或老化内容。推文没说有没有养号。

**合规。** TikTok 服务条款禁止自动化账号创建。报这条流程的账号能活多久,取决于推文没谈的那一层。这是一轮单跑操作;要长久跑,平台风险不小。

**地理围栏漂移。** 什么是"trending"取决于网络、设备指纹和账号历史。两次跑可能拿到不同的 trending 集。摘要每次都是非确定的。案例把 trending 当成稳定目标;实际上它在动。

**回退路径。** 任意一步失败,谁重试?agent 发 fallback 还是用户重发 iMessage?没有一份回退契约,瞬时失败会变成用户可见的挫败感。

## 这条回路哪里能用,哪里会断

适合:

- 用户被目标 App 地理封禁。
- 目标是单次一次性(注册、抓取、登出)。
- 失败可逆。
- 延迟容忍 1–10 分钟。

不能用于:

- 需要持续状态的任务(消息、发帖、回帖)。
- 用设备绑定的平台(大多银行 App)。
- 用户需要实时通知(回路是 pull)。
- 失败代价高(首次违规就失号)。

这些场景再迭代,要把 agent 带持久状态本地跑,而不是分发单条消息。报出来的 iMessage 流程是便宜的起步走法,不是稳态架构。
