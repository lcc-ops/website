Lovable,一支瑞典 AI 应用生成器,据披露在不到两年内做到 5 亿美金 ARR、不到 200 人、估值 66 亿美金。产品不训自己的模型,把第三方大模型包成「描述需求、得到一个能跑的全栈应用」流程。下面拆单位经济学,以及定价档位实际意味什么。

## 数字

| 数量 | 值 |
|---|---|
| 年化营收(最新) | 5 亿美金 |
| 员工数 | <200 |
| 人均营收 | 200 万美金+/年 |
| 估值 | 66 亿美金 |
| 单轮融资 | 3.3 亿美金 |
| ARR 最近一步增长 | 25%+ |
| 每周新项目数 | ~100 万 |
| 已上线项目月访问 | 6 亿 |
| 总用户数 | 800 万(过半用在能赚钱的活上) |

数字为公司自报、未审计。用作方向性论据,不是财报。

## 定价档位

案例点了三档。形状比具体数字重要:

```
免费    →  做出能用的成品,无额度
Pro     →  25 美金/月,100 个额度,每个项目能用的容量更大
Business→  50 美金/月,团队协作、角色权限、审计日志
```

计费单位是项目复杂度与产出数量。这是「先出货,再按用量计费」——和 Vercel、和云 egress、和所有用量计费的基础设施产品同形。先免费,用户的第一个项目免费,价格随用户实际在做的量往上走。

这个形状是承重决策,不是具体价格点。具体价格点以后可以调。「免费起步,再按用量扩张」这个形状,是非技术买家能安心接受的——在做出他们想继续做的东西之前,他们不付钱。

## 为什么客户是「非技术」而不是「开发者」

开发者把 Lovable 对标到他们已有的栈:VS Code、Git、一个 LLM 订阅。月成本大约 0–20 美金。在开发者已经具备这套肌肉记忆的前提下,Lovable 在任何价格点上都不便宜。

非技术买家把 Lovable 对标到外包报价。一个基础网站或应用 3000–30000 美金。Lovable 月费 25–50 美金。非技术买家没有「LLM 包出来的产品够不到」的价格上限。他们买的是「我不用去找、考核、付款给外包」,这在一个项目上就省下 5000 美金。

案例说得直接:护城河在非技术买家。开发者市场拥挤且价格敏感。非技术买家市场大、竞争小、结构上愿意付费,因为他们的替代方案是「项目根本不做」。

## 案例点出的飞轮

案例给出三层叠加:

1. **每周 100 万个新项目上线平台。** 每个项目默认公开,每个项目就是一个营销面。
2. **这些项目合计每月 6 亿次访问。** 访问者看到被做出来的东西,品牌与信任一起涨,反哺 SEO。
3. **800 万用户里过半用在能赚钱的活上。** 这不是闲着试玩的人。这是在平台上跑真实生意的人,他们会为 Pro 和 Business 付费。

飞轮是「项目数 → 访问数 → 用户数 → 付费用户数」。四个环节没有哪个是新的。新的是四个环节在同一平台上同时跑,因为每个环节都在反哺下一个,所以很难复制。

## 「不做模型」的姿态

Lovable 不训自己的基础模型。案例说得很直白:它包的是第三方大模型。竞争壁垒是「包」——项目结构、提示词到应用的流程、素材库、部署管线。任何竞争者都能调同样的 API,但很少有人能以同样节奏交付同样质量的包。

这笔交换和之前那支修图 App 一样:跳过训练基础模型的资本开支,背推理 API 的运营成本,把工程力量压在包上。赌的是「包」的累积速度跑赢模型的累积速度。两年下来,赌对了。

## 案例没说的事

案例把 Lovable 做了什么讲得具体,能不能复制这套数学,关键看四点:

1. **LLM API 之后的毛利。** 5 亿美金 ARR、每周 100 万个项目,推理账单不小。案例没给单位经济学表。推理成本是最大可变成本,也是最可能的毛利压缩项。
2. **留存曲线。** 做完一个项目的非技术用户,还会不会再做第二个?「先出货,再按用量计费」依赖用户回来。案例没给 6 个月或 12 个月的同期群留存。如果 Pro 订阅用户有一半做完一个项目就流失,模型就垮。
3. **合规风险。** AI 生成代码的版权与 IP 在大多数司法管辖区都没定论。欧盟或美国一个判例性的诉讼就可能逼着包一层重做。案例没点法律风险。
4. **多模型依赖。** Lovable 距离一次 API 中断导致部分宕机只差一次故障。案例没点名 failover 架构或推理在多模型路由上的比例。一次营销活动期间 24 小时宕机,可能直接砍掉可观营收。

定价档位的形状是可移植的。这套「包」能不能顶住一个认真的竞争者,案例没回答。
