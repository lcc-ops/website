HOW TO READ THE SOURCE ROW

The body text in the database is GBK-encoded bytes written to SQLite as latin1 codepoints. To get the real UTF-8 text, use this exact round-trip in any language or tool you have available:
    bytes_unicode.encode('latin1').decode('utf-8')

Read the row from the topics table whose id (zsxq topic_id) or tweet_id (x) equals ROW_ID. Use the decoded body_text and (for zsxq) the comments / (for x) the count columns as your source.

HARD RULES — read each one and follow strictly

1. NO network. NO browser. NO API calls. NO web fetch. Use only the SQLite row + your own knowledge of the operator-notes landscape.
2. NO attribution: no author name, no group handle, no platform URL, no "原帖", no "Source:" block, no "tweet id" mention. The post reads as if the writer is documenting the case directly.
3. NO meta content: no "as an AI", no signature, no Co-Authored-By, no "this post was generated by", no "I'll now write".
4. Frontmatter MUST be valid YAML wrapped in --- fences. Required keys, in order: title, description, pubDate, category, tags, translationKey, tldr, faq.
5. category MUST be exactly '{CATEGORY}' (single-quoted). Do NOT use 'ai'.
6. tags MUST be a 3-6 item array. The first tag MUST be the niche slug (e.g. 'logistics', 'payment-fees', 'google-ads'); remaining tags are concrete subject tokens (e.g. 'cross-border', 'cbm', 'fees', 'comparison'). Do NOT include 'case-study' or 'monetization' — those are AI-track markers only.
7. translationKey MUST equal the slug used in the output filename.
8. tldr MUST be a single quoted string, 200-450 chars, math-driven, no emoji.
9. faq MUST contain exactly 6 entries. Each entry has q (string) and a (string). Don't number them; don't prefix "Q:". Answers are 2-4 sentences each, self-contained.
10. BODY must contain exactly these 5 H2 sections, in this order. Any other H2 section is forbidden:

    ## <Numbers or comparison — a markdown table with units in the header row>
    ## <Cost structure / unit economics — a second markdown table or code block + prose>
    ## What this case does not cover
    ## Take-away
11. NO source footer. NO CTA. NO image markdown. NO callout block. NO emoji.
12. Body word count 1500-2200 across both languages. en and zh should be roughly matched in length.
13. en.md and zh.md must agree on numbers, niche, name of the case, and section structure. They are translations of the same case.
14. DO NOT introduce AI-tool framings (no "use Claude to...", no "automate with GPT...", no "LLM-driven..."). The case is about a real operator decision, not an AI workflow.
